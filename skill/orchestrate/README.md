# /orchestrate — оркестрация задач по моделям

Глобальный скилл Claude Code: основная сессия (Fable или Opus) выступает
оркестратором — декомпозирует задачу, раздаёт подзадачи агентам на разных моделях,
проверяет качество и синтезирует результат. Модели воркеров закреплены во фронтматтере
агентов и от модели-лидера не зависят. Работает целиком внутри Claude Code, в рамках
подписки.

## Состав

| Файл | Назначение |
|---|---|
| `SKILL.md` | Плейбук оркестратора: триаж → план → делегирование → проверка → синтез |
| `references/delegation.md` | Шаблон задачи-«тикета», примеры, шпаргалка промптинга под каждую модель |
| `references/quality.md` | Метрики качества: критерии приёмки, иерархия проверок, схема вердикта, скоркарта |
| `~/.claude/agents/architect.md` | Opus 4.8, effort xhigh — проектирование, сложный анализ, отладка |
| `~/.claude/agents/builder.md` | Sonnet 5, effort high — реализация по спецификации |
| `~/.claude/agents/scout.md` | Haiku 4.5 — механическая разведка; структурно read-only (только Grep/Glob/Read, без shell) |
| `~/.claude/agents/critic.md` | Opus 4.8, effort xhigh — адверсариальная проверка, свежий контекст |

## Использование

- `/orchestrate <задача>` — явный запуск.
- Автоматически: Claude сам подхватывает скилл на крупных многочастных задачах
  (описание скилла содержит триггер-фразы «оркеструй», «раздай задачи» и т. п.).
- Агентов можно звать и напрямую, без скилла: «спроси у architect…»,
  «пусть critic проверит…».

## Ключевые решения

- **Haiku — да, но узко**: только механические, однозначные, read-only тикеты с точным
  форматом вывода и стоп-правилом `NEEDS_CLARIFICATION` вместо догадок. Экономит квоту
  Opus/Fable; любая задача с суждением уходит на Sonnet. Read-only обеспечен
  структурно — у scout нет Bash и инструментов записи, а не только запретом в промпте.
- **Дисциплина рабочего дерева**: параллельные builder'ы пишут только в
  непересекающиеся пути; полные прогоны тестов — после завершения волны (у critic);
  критерии «diff только в разрешённых путях» проверяются по пофайловому снимку diff,
  а не по глобальному git status.
- **Производитель никогда не проверяет сам себя**: critic получает только результат и
  критерии приёмки, без рассуждений производителя (рекомендация Anthropic —
  fresh-context refutation).
- **Сложное — никогда вниз**: классы подзадач (суждение / исполнение по спецификации /
  механика) с жёсткими полами — архитектура, root-cause, безопасность, финальная
  проверка кода и содержательные линзы аудита идут только на Opus. При сомнении между
  уровнями — маршрут вверх: промах вверх стоит квоты, промах вниз — тихой ошибки.
  Эскалация односторонняя: провалившаяся на уровне подзадача вниз не возвращается.
- **Эскалация ограничена**: FAIL → один повтор с замечаниями → подъём уровня модели →
  доклад о блокере. Не более двух попыток на подзадачу.
- Скилл и агенты — англоязычные (промпты моделям), отчёт пользователю — на его языке.

## Changelog

- 2026-07-04 — v1.4: `effort: xhigh` во фронтматтере скилла — триаж, декомпозиция и
  роутинг у лида гарантированно идут на максимальной глубине рассуждения независимо от
  сессионного effort (пины воркеров были заданы раньше: architect/critic xhigh,
  builder high, scout — параметр не поддерживается Haiku).
- 2026-07-04 — v1.3.1: плейбук сделан независимым от модели-лидера — оркестратором
  может быть Fable или Opus; модели воркеров прибиты во фронтматтере и не наследуются;
  для Opus-лидера добавлено напоминание о его склонности недоделегировать.
- 2026-07-04 — v1.3: детерминированный роутинг по сложности. Рубрика «класс подзадачи →
  модель» (суждение → Opus, исполнение по спецификации → Sonnet, механика → Haiku);
  правило «при сомнении — вверх» (стоимость ошибок асимметрична); жёсткие полы —
  архитектура, root-cause, безопасность, финальная проверка кода, содержательные линзы
  аудита никогда не опускаются ниже Opus; смешанный класс — делить или брать высший;
  эскалация односторонняя; обязательный письменный routing-план с обоснованием каждого
  маршрута ниже Opus; ужесточены условия downgrade для critic (три условия
  одновременно, «экономия квоты» — не основание).
- 2026-07-04 — v1.2: обязательная маршрутизация в Workflow-сценариях — обнаружено на
  практике: `agent()` без `agentType`/`model` наследует модель сессии (Fable), и совет
  самого инструмента «model по умолчанию не указывать» ломал матрицу маршрутизации.
  Теперь каждый `agent()` обязан нести `agentType` (предпочтительно) или явный `model`;
  сценарий с «голыми» вызовами не запускается. Плюс фикс install.sh по вердикту critic:
  `--copy` теперь бэкапит существующую несимлинк-цель в `.bak`, а не удаляет её молча.
- 2026-07-04 — v1.1: адверсариальное ревью (3 линзы, 23 находки → 9 подтверждено).
  Исправлено: ложная атрибуция «goal, not procedure» для Opus 4.8 (это указание для
  Fable 5; Opus 4.8 следует инструкциям буквально — scope указывать явно); у scout
  отобран Bash (read-only теперь структурный); добавлена дисциплина рабочего дерева
  для параллельных builder'ов; протокол NEEDS_CLARIFICATION/stop-and-report включён
  в лестницу эскалации; уточнены цитаты источников (15×/4× стоимость токенов,
  «different model» для судьи, автоматическая инжекция CLAUDE.md субагентам).
- 2026-07-03 — v1.0: первая версия (скилл + 4 агента + 2 справочника).
