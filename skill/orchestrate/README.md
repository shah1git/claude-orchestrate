# /orchestrate — оркестрация задач по моделям

Глобальный скилл Claude Code: основная сессия (Fable или Opus) выступает
оркестратором — декомпозирует задачу, раздаёт подзадачи агентам на разных моделях,
проверяет качество и синтезирует результат. Модели воркеров закреплены во фронтматтере
агентов и от модели-лидера не зависят. По умолчанию самодостаточен и работает целиком
внутри Claude Code, в рамках подписки; при обнаруженных коннекторах включается
кросс-провайдерный слой (Codex, Gemini/Antigravity) — не-Claude-воркеры как регулярные
члены пула маршрутизации по решению лида (постоянный мандат 2026-07-09); без
коннекторов тихо деградирует до чистого Claude.

Модели указаны семействами, без номеров версий: во фронтматтере агентов стоят
плавающие псевдонимы (`fable`, `opus`, `sonnet`, `haiku`), и каждый запуск автоматически
использует новейшее поколение семейства. Версионно-специфичные заметки о поведении
моделей собраны в `references/delegation.md` под явным баннером «Verified for …»
с правилом перепроверки при смене поколения.

## Использование

- `/orchestrate <задача>` — явный запуск.
- Автоматически: Claude сам подхватывает скилл на крупных многочастных задачах
  (описание скилла содержит триггер-фразы «оркеструй», «раздай задачи» и т. п.).
- Агентов можно звать и напрямую, без скилла: «спроси у architect…»,
  «пусть critic проверит…».

## Ключевые решения

- **Haiku — да, но узко**: только механические, однозначные, read-only тикеты с точным
  форматом вывода и стоп-правилом `NEEDS_CLARIFICATION` вместо догадок. Экономит квоту
  Opus/Fable; любая задача с суждением уходит на Sonnet. Read-only обеспечен
  структурно — у scout нет Bash и инструментов записи, а не только запретом в промпте.
- **Дисциплина рабочего дерева**: параллельные builder'ы пишут только в
  непересекающиеся пути; полные прогоны тестов — после завершения волны (у critic);
  критерии «diff только в разрешённых путях» проверяются по пофайловому снимку diff,
  а не по глобальному git status.
- **Производитель никогда не проверяет сам себя**: critic получает только результат и
  критерии приёмки, без рассуждений производителя (рекомендация Anthropic —
  fresh-context refutation).
- **Сложное — никогда вниз**: классы подзадач (суждение / исполнение по спецификации /
  механика) с жёсткими полами — архитектура, root-cause, безопасность, финальная
  проверка кода и содержательные линзы аудита никогда не опускаются ниже Opus, а маршрут
  «архитектура/root-cause» с v2.3 идёт выше — на Fable-architect. Потолок маршрутизации
  вверх (`model: fable`) — применения перечислены в `routing.overrides.fable_ceiling_uses`;
  двойная Fable-линза корректности высших ставок из v2.3 упразднена вместе со всем
  механизмом профилей ставок (ADR-0002, конфиг v11). При сомнении между уровнями —
  маршрутизация вверх (`routing.tie_break`): промах вверх стоит квоты, промах вниз — тихой
  ошибки. Эскалация односторонняя: провалившаяся на уровне подзадача вниз не возвращается.
- **Эскалация ограничена**: FAIL → один повтор с замечаниями → подъём уровня модели
  (haiku → sonnet → opus → fable-воркер в свежем контексте → лид) → доклад о блокере.
  Не более двух попыток на подзадачу. Ступень fable-воркера существует потому, что
  «лид сделает сам» — не свежий взгляд: к этому моменту контекст лида уже несёт все
  провалившиеся попытки.
- **Рубрика калибруется данными**: каждый делегированный тикет фиксируется в
  `telemetry/routing-log.jsonl` (класс, маршрут, вердикт с первой попытки, эскалации);
  пороги пересмотра рубрики — quality.md §7. Маршрутизация перестаёт быть навсегда
  априорной.
- **Хребет Покока безусловен — Цена не масштабирует Метод** (ADR-0002,
  `docs/adr/0002-pocock-spine-unconditional.md`): критерий входа Задачи и ось `shape`
  (конфиг `shape.values`) заменили профили ставок (`low`/`standard`/`high`) — тривиальных
  Задач внутри оркестратора не бывает по построению. Сборка vs разбирательство и полная
  последовательность шагов — SKILL.md, «The build spine» и Step 0.
- **Два входа, хребет безусловен на обоих** (ADR-0004,
  `docs/adr/0004-frontier-entry-skill.md`): подготовительная фаза (грилл → спека →
  тикеты) может пройти в сольной сессии владельца, а исполнение готового фронтира
  тикетов берёт соседний тонкий скилл `/orchestrate-frontier` — свежее окно, страж
  провенанса (нет опубликованной спеки и восходящих к ней тикетов — отказ), общая
  машинерия в одном экземпляре. Этот скилл не переписан: полный вход остаётся для
  сырых задач и разбирательств.
- **Уточнение до декомпозиции**: перед дорогой волной лид проводит грилл-гейт (Step 0.5,
  references/clarify.md) — сократический допрос по одному вопросу за раз, число вопросов —
  суждение агента, а не квота; артефакты CONTEXT.md/ADR по умолчанию не пишутся. Глубина
  зависит от формы работы: на `разбирательство` — по триажу (таблица «тир × неоднозначность»),
  как раньше; на `сборка` грилл идёт полным и безусловным перед спекой (ADR-0002, спека #4).
  Плюс чекпойнт одобрения плана до первого воркера.
- **Аудит связующей ткани**: critic для сводок сверяет с источником кванторы и
  причинно-временные связки («всегда», «поэтому», «после») — там, где галлюцинируют сводки.
- **Гейт — фиксированные три Линзы, не масштаб** (ADR-0002, конфиг `gates.lenses`,
  CONTEXT.md «Линза»): Standards (хорошо ли сделано), Spec (то ли сделано, не лишнее ли),
  критик (можно ли верить; единственный вердикт) — одна волна, без масштаба по ставке и без
  триггеров; исполнители, состав и адъюдикация — SKILL.md Step 4, конфиг `gates.lenses`.
  Профили ставок (v3–v9) и кросс-модельная третья линза (v3–v10) упразднены вместе с осью
  `stakes:`.
- **Кросс-провайдер — решение лида под постоянным мандатом, две поверхности, с определением
  доступности**: при найденных коннекторах не-Claude-модели (Codex, Gemini/Antigravity) —
  регулярные члены пула маршрутизации (мандат 2026-07-09, v2.7; отменил opt-in-модель v2.0).
  Две постоянные ценности мандата: независимая аналитическая линза некоррелированной модели —
  начиная с конфига v11–v12/ADR-0002 это Standards-линза, штатный (не opt-in) член фиксированного
  трёхлинзового гейта на каждом гейтуемом деливерабле — и quota-spread (дефолт-лейны Use 4).
  Инварианты качества не сдвигаются: judgment-лейны и первичный грейдер — всегда Claude; маршрут без
  найденной поверхности тихо откатывается на Claude-воркера. **Единая модель вызова (v26): каждый
  внешний лейн — своя консольная утилита** (`codex exec`, `grok -p`, `agy`, `kimi -p`); ни моста,
  ни MCP в пути лейнов (ADR-0005). MCP-сервер Codex остаётся лишь ad-hoc «вторым мнением» вне гейта.
  Свой тонкий код (валидатор, писатель телеметрии, будущий `run-lane`) едет с репо; чужой (vendor
  CLI) — по наличию. Путь «только Claude» остаётся самодостаточным; «производитель ≠ проверяющий»
  выходит сам собой.
- **Плейбук учится на сбоях**: журнал инцидентов (quality.md §8) — качественная пара к
  количественной телеметрии §7; 2+ сбоя одного класса → правка правила, а не симптома.
  Плюс дисциплина спеки: проверка обязана уметь упасть (N1), никаких hedge-слов в тикете
  (N2), scout строго «глаза, не голова» — по списку глаголов (N3).
- **Fable — там, где концентрируется суждение, а не масса вызовов**: `architect` закреплён
  за Fable (проектирование, root-cause, риски — ~1 вызов на прогон, стоимость ограничена;
  тикет пишется «цель, а не шаги»); финальная ступень эскалации — другое санкционированное
  применение `model: fable` (`routing.overrides.fable_ceiling_uses`), не отдельная линза
  гейта. Двойная Fable-линза корректности высших ставок из v2.3 упразднена вместе с
  профилями ставок (ADR-0002); её задачу декорреляции отказов гейта теперь решает штатная
  Standards-линза на не-Claude исполнителе (см. выше). `builder` сознательно остаётся
  Sonnet: самая массовая роль, буквальное исполнение спеки — там буквальность достоинство,
  а фронтирная квота — расточительство.
- Скилл и агенты — англоязычные (промпты моделям), отчёт пользователю — на его языке.

  где видно, как работает оркестрация и какие ручки есть; слепок конфига — в телеметрию,
  «чтобы понимать глубину глубин»). Архитектурный принцип — один источник на значение:
  конфиг забирает у прозы ЗНАЧЕНИЯ (сигналы классов маршрутизации, жёсткие полы, лимиты
  лестницы эскалации, размеры вееров, триггеры двойной линзы, кросс-провайдерные пины и
  дефолт-лейны, пороги перекалибровки), проза оставляет метод и обоснования и ссылается
  на ключи; при конфликте побеждает конфиг. Исключение названо явно: пины четырёх
  Claude-агентов остаются во фронтматтере agents/*.md — там, где их реально исполняет
  харнесс; конфиг их не зеркалит (зеркало = гарантированный дрейф). Таблица пинов
  cross-provider.md переквалифицирована в запись о верификации с правилом «конфиг
  опережает баннер → сначала проверить по каталогу коннектора»; дедуплицированы Step 2
  (сигналы/полы), Step 4.3 (триггеры линз) и пороги quality.md §5/§7. Провенанс: каждая
  запись журнала штампуется отпечатком `config: "v<version>+<sha256:7>"`; при первом
  появлении нового отпечатка полный слепок ложится в `telemetry/config-snapshots/` —
  любая строка телеметрии разворачивается в точные байты конфига, её породившего, даже
  вне git-истории; перекалибровка теперь отличает «рубрика ошибается» от «рубрика
  сменилась посреди выборки». Новое: политика недоступности (Step 2 «Availability
  fallbacks», секция `availability` конфига) — недоступность модели (окно лимитов,
  неоплаченная подписка, отсутствующий/разлогиненный коннектор, ошибка поверхности) —
  событие доступности, а не качества: попыток лестницы эскалации не расходует; рерут идёт
  по редактируемой цепочке (первый доступный побеждает) с правилом направления «никогда
  ниже пола качества класса — только вбок к равному тиру другого провайдера или вверх»;
  первичный грейдер остаётся Claude всегда; терминальные элементы — честные поведения
  (report-blocker / lead-inline / skip-third-lens), не подмена модели; сработавший рерут
  виден в телеметрии полями fallback_from/fallback_reason — давление квоты становится
  данными, а не тихо перекраивает маршрутизацию. В конфиге — раздел-карта «кандидаты на
  будущие ручки» (глубина грилла, чекпойнт одобрения, бюджет сессии и др.); словарь
  вердиктов §7 сознательно НЕ ручка — стабильность делает записи разных недель
  сравнимыми.
