# Ужесточение Kimi Code CLI

Основание — егресс-аудит открытого монорепо `MoonshotAI/kimi-code` от 2026-07-18
(архитектор). Транспорт Kimi у нас — **немодифицированный официальный CLI**
(`kimi -p` headless, OAuth-подписка Allegretto, модель k3, 1M), драйвинг через
терминал Orca; идентификатор клиента (User-Agent, `X-Msh-*`) не трогаем — это
требование ToS и техническая необходимость (без них бекенд отвечает 40340).

## Ключевое отличие от Grok

**Молчаливой выгрузки репозитория, как у Grok, НЕТ.** Единственная упаковка
кодовой базы — третья ступень команды `/feedback` (явный выбор «No / Logs only /
Logs + codebase», без предвыбора), в headless недостижима. Автоматических
снапшотов, «cloud sync», бакетов не найдено. Идентификатор git-remote в телеметрию
сейчас не уходит (функция `normalizeRemote` — мёртвый экспорт; отслеживать при
обновлениях версии).

## Пять штатных каналов наружу и как их сузить

| Канал | По умолчанию | Ручка |
|---|---|---|
| Инференс `api.kimi.com/coding/v1` | — | нужен; payload = промпт+диалог+результаты инструментов; в заголовках hostname+device_id |
| Телеметрия `telemetry-logs.kimi.com` (только метаданные, но с OAuth-токеном → связываемо с аккаунтом) | **включена** | `telemetry = false` в config.toml + env `KIMI_DISABLE_TELEMETRY=1` |
| Auto-update `code.kimi.com` (проверка + **фоновая установка, даже headless**) | **включено** | env `KIMI_CODE_NO_AUTO_UPDATE=1` |
| Серверные search/fetch через бекенд Moonshot | **логин прописывает сам** | удалить `[services.moonshot_search]` / `[services.moonshot_fetch]` (fetch станет локальным); перезаписывается при каждом re-login |
| `/feedback` выгрузка (даже «Logs only» = wire events = полный диалог) | по команде | **не использовать `/feedback` в этом CLI** |
| Cron-инструмент | включён | env `KIMI_DISABLE_CRON=1` |

## Что применяем и где

**Глобально (живой `~/.kimi-code/config.toml`, безопасно — без потери функций):**
```toml
telemetry = false   # применено 2026-07-19; privacy, поведенческие события off
```

**Env спавнера лейна (когда `/orchestrate` запускает kimi как воркер — не трогает
интерактивные сессии владельца в omp/нативном CLI):**
```bash
export KIMI_DISABLE_TELEMETRY=1     # ремень к telemetry=false + гасит хвост failed_*.jsonl (живёт 7 дней)
export KIMI_CODE_NO_AUTO_UPDATE=1   # ни проверки, ни фоновой установки в headless
export KIMI_DISABLE_CRON=1          # recon-лейну расписания не нужны
```

**Опционально, осознанное решение владельца (НЕ применено по умолчанию — трогает
интерактивный веб-поиск):** удалить `[services.moonshot_search]` и
`[services.moonshot_fetch]` из `config.toml`, если не хотим, чтобы поисковые
запросы и загружаемые URL шли через бекенд Moonshot. Для recon-лейна веб-поиск не
нужен → в лейне можно убрать; для общих интерактивных сессий владельца — оставить,
если поиск полезен. Секция восстанавливается при каждом login/refresh — если
убирать, повторять после каждого re-login.

## Изоляция

У Kimi CLI **нет** собственной файловой песочницы: политика путей
`absolute-outside-allowed` (любой абсолютный путь вне workspace читаем при
авто-одобрении), канонизация лексическая (симлинк наружу границу обходит). Блок-лист
чувствительных файлов мал (`.env*`, `id_rsa`/`id_ed25519`/…, AWS/GCP creds). Вывод:
**worktree — гигиена того, что уходит само, но НЕ барьер чтения.** Жёсткую границу
даёт только OS-уровневая изоляция — гнать под пользователем/в контейнере Orca, где
за пределами worktree нет ничего чувствительного. В системный промпт автоматически
уходят `AGENTS.md` + листинг имён файлов cwd; в заголовках каждого запроса —
hostname машины (при чувствительности — нейтральный hostname контейнера).

## Правило эксплуатации

`/feedback` в этом CLI не использовать никогда — даже «Logs only» выгружает wire
events (полное содержимое сессии).
