# Дисциплина RECON — большеконтекстная разведка

Добавлена 2026-07-18 под кандидата **kimi-recon** (Kimi Code CLI, k3, 1M контекст,
Allegretto-подписка, OAuth). Вопрос дисциплины — тот же, что у полигона в целом
(DESIGN §1): *берёт ли кандидат планку роли там, где закреплённые/дешёвые лейны
её теряют.* Ниша, ради которой Kimi вообще рассматривается, — **инвентаризация
корпуса, который не влезает в скромное окно контекста.**

## Что меряем

Механическая инвентаризация мест `raise ReconError(...)` в синтетическом корпусе
(та же дисциплина, что scout t1: точный ключ, precision/recall ≥ 0.95). Единственная
переменная — **размер входа**:

| Тир | Вход | Кто должен держать |
|-----|------|--------------------|
| t1  | ~40 файлов / ~42K токенов  | все лейны (калибровка) |
| t2  | ~200 файлов / ~200K токенов | давит haiku/sonnet (~200K окно) |
| t3  | ~600 файлов / ~600–700K токенов | превышает luna (372K); держат только 1M (k3, gemini-flash) |

Выход умеренный (48 / 200 / 450 маркеров) специально — чтобы лимит **выходных**
токенов не спутать с потерей **входного** контекста.

## Кандидаты (лейны, не «голые» модели)

- **kimi-recon** (испытуемый): kimi-code k3, effort high, **транспорт B3 — терминал
  Orca** (немодифицированный официальный CLI + родная OAuth; НЕ наш bridge — ToS
  запрещает подмену идентификатора), в выделенном worktree, с ужесточённым
  конфигом из аудита (см. references/grok/ как образец; kimi-аудит в работе).
- **gemini-recon** (реальный конкурент за ту же нишу): Gemini 3.5 Flash, ~1M —
  единственный второй 1M-лейн, что у нас уже есть. Прямое сравнение k3 vs
  gemini-flash на t3 — главный практический вопрос: даёт ли Kimi то, чего Gemini
  не даёт.
- **Якоря/инкумбенты:** scout (haiku, ~200K — нижний якорь, ожидаемо роняет на
  t2/t3), codex-recon (luna gpt-5.6, 372K — роняет на t3), sonnet (200K, up-tier).

Критиком Kimi **никогда** (reward-hacking-класс непроверенных моделей; как Grok).

## Гипотеза и решение

Гипотеза: на t3 планку ≥0.95 держат только k3 и gemini-flash; остальные роняют
recall (не удержали корпус). Если k3 ≈ gemini-flash — Kimi дублирует, нишу не
оправдывает. Если k3 > gemini-flash (по точности/стоимости/скорости) — оправдывает
recon-лейн. ≥10 записей в общий репо телеметрии до любой промоции (config
`cross_provider.pilots`).

## Как прогонять

```sh
# 1. Ключи (детерминированы генератором; истина записана ДО прогонов):
python3 tools/gen_recon_corpus.py gen
POLYGON_PASS=… tools/seal_keys.sh seal      # запечатать перед прогонами

# 2. На КАЖДОГО кандидата — свежий staged workspace с материализованным корпусом:
python3 tools/gen_recon_corpus.py corpus t3 /path/to/staged
#   -> /path/to/staged/corpus/*.py ; выдать кандидату ticket.md этого тира

# 3. Грейд снятого с диска ответа кандидата:
python3 tools/gen_recon_corpus.py grade t3 /path/to/candidate-answer.json

# 4. После всех прогонов — распечатать ключи для разбора:
POLYGON_PASS=… tools/seal_keys.sh unseal
```

**Гигиена ключей (DESIGN §4):** открытый текст `tasks/recon/*/hidden/key.json` в
git не коммитится — только `hidden.sha256` + `hidden.tar.enc` после `seal`.
Корпус в git не хранится вовсе: он детерминированно материализуется генератором
в staged workspace (объём ~2.4MB на t3).

**Безопасность прогона:** корпус полностью синтетический (ноль реальных/клиентских
данных), поэтому recon-полигон можно гонять на Kimi **до** завершения полного
егресс-аудита CLI — наружу уходит только синтетика. Аудит гейтит боевую работу с
реальным кодом, не этот полигон.
