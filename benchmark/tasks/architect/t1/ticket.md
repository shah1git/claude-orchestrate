# architect t1 — диагностика: повторное бронирование неосмотренного прибора

## OBJECTIVE
В выданной копии проекта `depot` есть один подсаженный дефект. Определи его
**первопричину**: конкретную строку кода, механизм, из-за которого она
производит наблюдаемый симптом, и причинную цепочку от этой строки до
симптома. Возвращать исправление не нужно — только диагноз.

## CONTEXT
Домен — учёт и бронирование геодезического оборудования (см. `README.md`
в корне выданной фикстуры). Каждый физический прибор проходит после
возврата цикл карантина, прежде чем его можно забронировать снова:

```
RECEIVED -> PENDING_CHECK -> CLEARED                        (осмотр пройден)
RECEIVED -> PENDING_CHECK -> QUARANTINE_ACTIVE -> CLEARED    (осмотр провален)
```

Правило склада зафиксировано в докстринге `InstrumentState`
(`depot/models.py`): «Only a CLEARED instrument may be booked» — то есть
разрешено бронировать **только** прибор в состоянии `CLEARED`. Прибор, только
что возвращённый и ещё не осмотренный, находится в состоянии `RECEIVED` и
бронироваться не должен.

## SYMPTOM
Прибор, который был выдан по одной брони, возвращён (`return_booking`), но
ни разу не отправлен на осмотр (`begin_inspection` / `resolve_inspection`
ни разу не вызывались — он так и остаётся в состоянии `RECEIVED`),
**успешно** принимается системой во вторую бронь. Симптом воспроизводится
детерминированно на выданной копии фикстуры.

## HOW TO RUN THE REPRO
В корне выданной рабочей копии (там, где лежит каталог `depot/`) выполни:

```
python3 repro.py
```

Скрипт: регистрирует прибор, проводит его через полный цикл
резервирование → выдача → возврат (оставляя его в `RECEIVED`, неосмотренным),
затем пытается создать вторую бронь на тот же прибор и печатает результат.
Строка, начинающаяся с `SYMPTOM:`, показывает, что вторая бронь **прошла**
(с идентификатором брони и состоянием прибора) — при том что по правилу
выше она должна была быть отклонена.

## ESCAPE HATCH
Если `repro.py` не запускается как описано (нет каталога `depot/`,
скрипт падает не на демонстрации симптома, а на импорте/окружении) — не
угадывай причину дефекта по догадке. Верни строку
`NEEDS_CLARIFICATION: <что ты увидел вместо ожидаемого>` и ничего больше.
