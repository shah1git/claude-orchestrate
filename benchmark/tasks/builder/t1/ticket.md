# builder t1 — новый модуль: залог за оборудование (deposit)

## OBJECTIVE
Реализовать новый модуль `depot/deposit.py`, вычисляющий размер залога за
комплект оборудования при бронировании и сумму возврата залога при сдаче
оборудования обратно.

## CONTEXT
Склад берёт с арендатора залог, покрывающий риск повреждения или утраты
оборудования. Залог не может быть меньше фиксированного минимума для класса
инструмента (мелкая, дешёвая по итоговой стоимости бронь всё равно рискует
дорогим прибором) и не может быть меньше доли от итоговой стоимости брони
(дорогая бронь пропорционально увеличивает риск). Действует больший из этих
двух порогов, но не выше фиксированного потолка для класса. При возврате из
залога вычитается сумма, поставленная в вину арендатору за повреждение;
результат не может уйти в отрицательную область — избыток ущерба сверх
залога взыскивается отдельно, вне этого модуля.

Проект уже реализует правило округления денег: `decimal.Decimal`,
округление до копеек (`ROUND_HALF_UP`) выполняется **один раз**, в самом
конце публичной функции — см. комментарий о округлении в начале
`depot/tariff.py` и его применение в `depot/booking.py`. Модуль `deposit.py`
следует тому же правилу и той же манере кода: докстринги в стиле модуля,
однострочные `raise DepotError(...)` там, где это принято в проекте (см.
`depot/tariff.py`, `depot/booking.py`).

## INPUTS
- Прочитай `depot/tariff.py`, `depot/booking.py`, `depot/models.py`,
  `depot/errors.py`, чтобы понять стиль, форму `DepotError` и правило
  округления.
- Пять классов инструментов заданы в `depot.models.InstrumentClass`:
  `THEODOLITE`, `GNSS_ROVER`, `LEVEL`, `TOTAL_STATION`, `DRONE_PHOTO`.
- Существующие коды `DepotError` в проекте — `DEP-001`..`DEP-025`. Новые
  коды этого модуля должны их не пересекать: используй `DEP-101`, `DEP-102`,
  … (диапазон `DEP-101`..`DEP-1xx`), по одному уникальному коду на каждое
  условие ошибки.

## OUTPUT
Новый файл `depot/deposit.py`, только стандартная библиотека + `Decimal`,
без сторонних зависимостей, со следующим публичным контрактом:

**Константы уровня модуля** (все — `Decimal`, по образцу префиксов
`RATE_`/`COEF_`/`CAP_` в `tariff.py`). Значения **заданы точно** — приёмочные
тесты сверяют вычисления именно с этими числами, поэтому используй их как есть,
с этими именами:

```
DEP_BASE_THEODOLITE    = Decimal("3000.00")
DEP_BASE_GNSS_ROVER    = Decimal("8000.00")
DEP_BASE_LEVEL         = Decimal("1500.00")
DEP_BASE_TOTAL_STATION = Decimal("6000.00")
DEP_BASE_DRONE_PHOTO   = Decimal("10000.00")

DEP_RATE_OF_TOTAL      = Decimal("0.30")

DEP_CAP_THEODOLITE     = Decimal("15000.00")
DEP_CAP_GNSS_ROVER     = Decimal("35000.00")
DEP_CAP_LEVEL          = Decimal("9000.00")
DEP_CAP_TOTAL_STATION  = Decimal("28000.00")
DEP_CAP_DRONE_PHOTO    = Decimal("42000.00")
```

Задача — не подобрать числа, а корректно реализовать логику (`max` двух полов,
зажатие потолком, однократное округление, обработка ошибок) поверх этих
заданных величин.

**`compute_deposit(instrument_class, booking_total: Decimal) -> Decimal`**
- `deposit = max(базовый_залог_класса, DEP_RATE_OF_TOTAL * booking_total)`,
  затем зажать сверху потолком класса (`min(..., потолок_класса)`), затем
  округлить до копеек `ROUND_HALF_UP` **один раз**, в самом конце.
- Возбудить `DepotError` при неизвестном классе инструмента и при
  отрицательном `booking_total`.

**`deposit_refund(deposit: Decimal, damage_surcharge: Decimal) -> Decimal`**
- `refund = deposit - damage_surcharge`, с полом в `Decimal("0.00")`
  (результат никогда не уходит в минус); округлить до копеек
  `ROUND_HALF_UP` один раз, в конце.
- Возбудить `DepotError` при отрицательном `deposit` и при отрицательном
  `damage_surcharge`.

Оформление: докстринги на уровне модуля и функций в манере
`tariff.py`/`booking.py`; однострочные `raise DepotError("DEP-1##", "...")`
там, где условие проверяется в одну строку (как в `tariff.compute_total`);
подробные комментарии там, где логика не самоочевидна.
