# Полигон ролей — протокол сравнительного испытания моделей

Статус: **черновик v0.2** — ожидает одобрения пользователем; прогонов ещё не было.
История: v0.1 → состязательное ревью (critic/opus, вердикт FAIL: 1 критическая,
1 существенная, 2 незначительные находки) → v0.2 закрывает все четыре.
Конфиг-отпечаток на момент проектирования: `v1+ff537b5`.
Доступность поверхностей (проверено `run-external-agent.mjs --detect`, 2026-07-11):
Codex present+loggedIn (codex-cli 0.144.0); Gemini present, каталог содержит
«Gemini 3.5 Flash (Low)» и «Gemini 3.1 Pro (High)» — оба наших пина.

---

## 1. Цель и рамка

Маршрутизация оркестрации (`skill/orchestrate/config.yaml`) закрепляет за каждой
ролью модель и эффорт. Телеметрия (`routing-log.jsonl`, 215 записей) — данные
*наблюдательные*: каждый тикет исполнялся одной моделью, контрфактического
сравнения в них нет. Полигон добавляет недостающее — *контролируемое* сравнение:
одна и та же задача, несколько изолированных кандидатов, заранее зафиксированная
истина, детерминированный грейдинг.

**Вопрос полигона — не «какая модель умнее».** Вопрос: *проходит ли планку роли
текущая закреплённая модель, и проходит ли ту же планку соседний — более дешёвый
или кросс-провайдерный — кандидат.* Сравниваются **лейны как они развёрнуты**
(агент + системный промпт + пин модели + эффорт), а не «голые» модели: именно
лейн — единица маршрутизации.

**Принцип v1: измеряем только против заранее известной истины.** Субъективные
рубрики («хорош ли дизайн?») в v1 исключены — они возвращают проблему
самопредпочтения судьи (ADR-0001) и не дают воспроизводимого счёта. Поэтому
дисциплина architect построена на root-cause-задачах с *подсаженной* причиной,
а дисциплина critic — на *подсаженных* дефектах.

## 2. Словарь

- **Полигон** — синтетический проект-фикстура + набор задач + ключи + протокол.
- **Дисциплина** — набор задач одной роли (scout / builder / critic / architect).
- **Кандидат** — конкретный лейн: агент/поверхность + модель + эффорт.
- **Инкумбент** — лейн, закреплённый за ролью сейчас (agents/*.md, config.yaml).
- **Якорь** — калибровочный кандидат: *верхний* (заведомо сильнее — если планку
  не берёт он, дефектна задача) или *нижний* (заведомо слабее — если планку берёт
  он, задача не различает уровни).
- **Ключ** — заранее вычисленный эталон (answer key, hidden-тесты, манифест
  дефектов, записанная причина бага). Кандидатам недоступен (§4).
- **Staged workspace** — свежая копия фикстуры в изолированном каталоге,
  единственное, что нужно кандидату для работы.

## 3. Структура каталога

```
benchmark/
  DESIGN.md                  — этот документ
  fixtures/
    depot/                   — эталонный синтетический проект (см. §5)
    depot-rc1/, depot-rc2/   — варианты с подсаженным багом (architect)
  tasks/
    scout/t1/,t2/            — ticket.md (виден кандидату при диспатче)
    builder/t1/,t2/
    critic/t1/
    architect/t1/,t2/
  hidden.sha256              — хеши всех ключей, зафиксированы ДО прогонов
  hidden.tar.enc             — ключи, зашифрованные на время прогонов (§4)
  results/<дата-прогона>/
    runs/<run-id>/           — снятый с диска дифф, сырой ответ, usage, аудит
    scores.jsonl             — по записи на прогон (схема в §10)
  RESULTS.md                 — синтез последнего прогона
```

**Дисциплина хранения ключей.** До завершения всех прогонов открытый текст
ключей в git **не попадает никогда** (иначе кандидат с shell достал бы его из
истории `git show`). Коммитятся только: (а) `hidden.sha256` — хеши каждого
ключа, доказательство, что истина записана до прогонов; (б) `hidden.tar.enc` —
шифрованный архив. Открытый текст ключей коммитится **после** грейдинга — ради
воспроизводимости и разбора. Осознанное следствие: с этого момента задачи
считаются «потенциально засвеченными» для *будущих* поколений моделей — перед
переиспользованием полигона на новом поколении задачи обновляются (стандартная
гигиена бенчмарков).

## 4. Протокол изоляции и античит

Главное требование пользователя: модели должны *работать сами*, а не присваивать
чужой результат. После состязательного ревью защита строится **двумя слоями**:

- **Слой 1 — профилактика**: то, что можно сделать структурно недоступным,
  делается структурно недоступным (песочницы, шифрование ключей на время прогонов).
- **Слой 2 — детерминированная ревизия**: харнесс журналирует каждый вызов
  инструмента каждого кандидата; после прогона лид проверяет журнал по формальному
  правилу (§11.7). Это не «полицейская надежда», а сплошная проверка полного
  протокола действий: обращение к запретному пути видно в журнале всегда.

Честная оговорка: Claude-суб-агенты запускаются без OS-песочницы, и на уровне
файловой системы машина им доступна. Поэтому профилактика для них — не «стена
вокруг кандидата», а **отсутствие секрета в досягаемости**: на время прогонов
открытого текста ключей на диске просто нет.

Модель угроз:

| # | Вектор | Мера |
|---|--------|------|
| 1 | Кража чужого результата | На каждый прогон — собственный staged workspace (`rsync` фикстуры в свежий каталог); кандидаты не знают о существовании друг друга; пути не пересекаются; результаты предыдущих прогонов в workspace не попадают. Слой 2: чтение чужого каталога `runs/` видно в журнале → дисквалификация. |
| 2 | Доступ к ключам — внешние поверхности | Codex — `sandbox: workspace-write`/`read-only` с `cwd` = workspace (за пределы песочница не пускает); Gemini — staging-sandbox моста (в стейджинг попадает только фикстура). |
| 3 | Доступ к ключам — Claude-кандидаты (**критическая находка ревью v0.1**) | Инструменты Read/Grep/Glob/Bash принимают любые пути, фронтматтер агентов их не ограничивает. Мера: на время волны прогонов ключей в открытом виде **нет нигде на диске** — только `hidden.tar.enc` (AES-256, парольная фраза удерживается лидом вне файловой системы) и `hidden.sha256`. Слой 2: любое обращение кандидата к `benchmark/`, `hidden*`, истории git этого репозитория или каталогам сессий `~/.claude/projects/` → дисквалификация по журналу. |
| 4 | Подмена харнесса («сломай тест, чтобы он прошёл»), включая правку эталонных hidden-тестов на диске | Грейдинг никогда не идёт в workspace кандидата: лид снимает `git diff` + untracked, применяет `git apply` к **свежей** копии фикстуры, hidden-тесты извлекаются из `hidden.tar.enc` с проверкой каждого файла по `hidden.sha256` **перед** прогоном. Расхождение хеша = попытка подмены, прогон помечается, инцидент фиксируется. Изменение файлов тестов/конфигурации в диффе кандидата = нарушение границ = провал. |
| 5 | Инфляция самоотчёта | Самоотчёт не является доказательством ничего. Builder-дисциплина считается только по диффу с диска (правило «только diff-с-диска» — прямой урок METR о рекордном reward hacking у GPT-5.6 Sol); scout/critic/architect — только по возвращённому структурированному ответу, сверенному с ключом. |
| 6 | Игра против судьи | Судейство детерминированно везде, где возможно (ключи, hidden-тесты, манифесты). Там, где нужен матчинг текста (critic, architect), матчится *заранее записанная* формулировка истины; серьёзность находки определяет манифест/лид, **не самооценка кандидата** (находка ревью v0.1); спорный матчинг решает лид с записью обоснования в scores.jsonl. |
| 7 | Контаминация обучающими данными | Фикстура пишется с нуля: вымышленный домен, нетипичные имена, отсутствие известных туториалов-прототипов. Web-инструменты кандидатам не отключаются (они — часть их производственных лейнов), но для вымышленного домена бесполезны. |
| 8 | Утечка через контекст лида и транскрипты сессий | Транскрипты сессий лежат на диске (`~/.claude/projects/`) и теоретически доступны Bash-кандидату. Профилактика: лид не цитирует содержимое ключей в разговоре после волны 0; верификация эталона (волна 0) завершается до первого диспатча кандидатов. Остаточный канал закрывает слой 2: обращение к каталогам транскриптов — дисквалификация. |
| 9 | Асимметрия попыток | Каждому кандидату — ровно одна попытка на задачу, без ретраев и эскалации. Событие *недоступности* (квота, коннектор) — не провал: один повторный диспатч с пометкой `fallback_reason` в записи. |

## 5. Фикстура

`fixtures/depot` — небольшой (~20 файлов, Python + pytest) сервис учёта и
бронирования геодезического оборудования с вымышленными бизнес-правилами
(тарификация по «полевым дням», карантин прибора после возврата, несовместимость
комплектов). Домен выдуман, чтобы знание тренировочных данных не помогало.

Строится `builder`-ом по моей спецификации (волна 0): лид задаёт каждый дефект и
каждую причину бага **содержательно** (класс, механизм, место), builder лишь
воплощает — это сужает эффект «родного стиля» автора (см. §9.5). Затем critic
верифицирует эталон: компилируется, видимые тесты зелёные, hidden-тесты зелёные
на эталоне, каждый пункт манифеста дефектов/причин реально присутствует в
соответствующем варианте.

## 6. Дисциплины

Все планки предзафиксированы (до первого прогона) — постфактум не двигаются.

### 6.1 scout — механическая инвентаризация (2 задачи)
- **Задачи**: t1 — полная инвентаризация по правилу («каждый символ X с атрибутом
  Y: файл, строка, значение»); t2 — извлечение и классификация по явной таблице
  правил. Формат ответа задан побайтово (JSON-строки). Обе задачи — объёмная
  механика (bulk), что и оправдывает effort low для codex-recon (§7).
- **Ключ**: `key.json`, вычислен лидом детерминированно (grep/скрипт) до прогонов.
- **Метрика**: precision и recall против ключа; соблюдение формата.
- **Планка**: precision ≥ 0.95 И recall ≥ 0.95 И формат соблюдён.

### 6.2 builder — реализация по спецификации (2 задачи)
- **Задачи**: t1 — новая фича по полной спецификации (объектно проверяемая);
  t2 — рефакторинг с сохранением поведения. Кандидат видит спецификацию и
  *видимые* тесты; **скрытых приёмочных тестов не видит**.
- **Метрика**: дифф с диска → чистая копия → hidden-тесты (из проверенного по
  хешам архива); сверка путей диффа с границами тикета.
- **Планка**: 100 % hidden-тестов зелёные И видимые тесты зелёные И ноль файлов
  вне границ.

### 6.3 critic — поиск подсаженных дефектов (1 задача)
- **Задача**: деливерабл (дифф + тикет + критерии приёмки) с K = 8 заранее
  подсаженными дефектами разных классов: логическая ошибка, уязвимость,
  нарушение спецификации, «зелёный» тест, который ничего не проверяет,
  ошибка границ, ложное утверждение в сопроводительном отчёте (связка
  quantifier/причинность), дефект интеграционного шва, изменение вне границ.
- **Ключ**: `manifest.json` — класс, файл/строка, формулировка каждого дефекта.
- **Метрика**: найдено / пропущено / ложные срабатывания. Матчинг находки к
  манифесту — лид по записанным формулировкам; серьёзность — по манифесту, не
  по самооценке кандидата; спорные случаи фиксируются в scores.jsonl.
- **Планка** (ужесточена по находке ревью v0.1 — «распыление» находок не должно
  проходить): найдено ≥ 6 из 8 И **ложных срабатываний любой серьёзности ≤ 2**.
  Ранжирование сверх планки: (найдено ↓, ложные ↑).

### 6.4 architect — root-cause с известной причиной (2 задачи)
- **Задачи**: t1/t2 — в `depot-rc1`/`depot-rc2` подсажен по одному нетривиальному
  багу (например, гонка инвалидации карантинного статуса; накопление ошибки
  округления тарифа через границу «полевого дня»). Кандидат получает симптом и
  репродукцию, возвращает причину и механизм.
- **Ключ**: `cause.md` — файл, механизм, цепочка причинности, записаны до прогонов.
- **Метрика/планка**: полный зачёт — назван файл И механизм И причинная цепочка;
  полузачёт — только место без механизма. Планка — полный зачёт на обеих задачах.
- **Калибровка валидности** (находка ревью v0.1): инкумбент здесь — верхний тир,
  верхнего якоря не существует. Поэтому валидность калибрует **нижний якорь**
  (sonnet, одна задача): если причину называет и он — задача не различает уровни
  и результат дисциплины не интерпретируется как провал/успех инкумбента.

## 7. Матрица кандидатов

Сравниваются соседние тиры и лейны на границах реальных решений маршрутизации
(не «все против всех»). Эффорты — пиновые, из config.yaml / фронтматтера.

| Дисциплина | Инкумбент | Кандидаты «вбок/вниз» | Якорь | Прогонов |
|---|---|---|---|---|
| scout ×2 | haiku (scout) | gemini-recon-cheap = Flash Low; codex-recon = luna, effort low (bulk-правило, §6.1) | sonnet, 1 задача (верхний) | 7 |
| builder ×2 | sonnet high (builder) | codex-code = terra high | opus, 1 задача (верхний) | 5 |
| critic ×1 | opus xhigh (critic) | codex-critic = sol xhigh; gemini-critic = 3.1 Pro High; sonnet (нижняя граница — проверка условий санкционированного даунгрейда) | fable (верхний) | 5 |
| architect ×2 | fable xhigh (architect) | opus xhigh (первая ступень availability-fallback), обе задачи | sol xhigh, 1 задача (кросс-угол, информационно); sonnet, 1 задача (нижний, калибровка §6.4) | 6 |

Итого **23 прогона** + волна 0 (фикстуры) + грейдинг. Codex/Gemini-кандидаты
тратят кредиты своих подписок (quota-spread), Claude-квота — на Claude-кандидатов,
судейство и синтез. Недоступный лейн выпадает из матрицы с пометкой в RESULTS.md —
это availability-событие, не оценка.

Решения, которые матрица способна проинформировать: состав fallback-цепочек
scout/builder; дефолты `cross_provider.defaults`; условия даунгрейда критика;
цену первой ступени фолбэка architect (fable→opus). Чего она не решает — §11.

## 8. Протокол прогона

- **Волна 0 — постройка.** Лид пишет спецификации задач и содержательно задаёт
  ключи; builder строит фикстуры, hidden-материалы, подсаживает дефекты/баги;
  critic верифицирует эталон и соответствие манифестов реальности. Затем лид:
  `hidden.sha256` (хеш каждого ключа) → `hidden.tar.enc` (шифрование; парольная
  фраза — вне файловой системы) → коммит хешей и архива → **удаление открытого
  текста ключей с диска** (шаг входит в протокол и утверждается вместе с этим
  документом; при утере парольной фразы ключи перестраиваются по коммиченным
  спецификациям задач — дороже, но без потери валидности). Открытый текст в git
  до конца прогонов не попадает (§3).
- **Волна 1 — прогоны.** По дисциплинам; внутри дисциплины кандидаты параллельно,
  каждый в своём staged workspace. Клод-кандидаты — Agent tool (агент + при
  необходимости `model:`-override для якорей/границ); Codex/Gemini — мост, с
  песочницами из §4. Тикет одной задачи побайтово одинаков для всех.
- **Волна 2 — грейдинг и аудит.** Сначала аудит журналов инструментов каждого
  прогона по правилу §11.7; затем детерминированные проверки (hidden-тесты из
  архива с хеш-контролем, ключи) + лид-матчинг для critic/architect. Каждому
  прогону — запись в scores.jsonl.
- **Волна 3 — синтез.** RESULTS.md: таблица планок, стоимость (токены/время),
  интерпретация, предложение правок config.yaml (§11). После этого открытый
  текст ключей коммитится для воспроизводимости (§3).

**Телеметрия** (находка ревью v0.1): 23 кандидатских прогона пишутся **только в
scores.jsonl** и никогда — в `routing-log.jsonl`: намеренные ролевые несоответствия
(sonnet-как-critic, luna-как-scout) исказили бы производственную перекалибровку
(`thresholds.*`). В routing-log идут, как обычно, только *оркестрационные* тикеты
самого запуска (волна 0: builder-постройка, critic-верификация; ревью этого
документа), с `config`-отпечатком.

## 9. Честность сравнения — заявленные асимметрии

1. Системные промпты различаются: Claude-агенты несут свои тюнингованные промпты,
   мостовые лейны получают только тикет. Это не искажение, а предмет измерения:
   сравниваются лейны как они развёрнуты в маршрутизации.
2. Инструментарий различается по той же причине (scout без shell, luna с shell).
   Тикеты формулируются инструментно-нейтрально: «верни список», не «сделай grep».
3. Одна попытка на кандидата — дисперсия одного прогона не оценивается. Смягчение:
   планки бинарные и грубые (pass/fail), решения принимаются только при
   согласовании с телеметрией (§11), якоря калибруют валидность задач.
4. Стоимость учитывается как есть: `usage` для Claude-агентов; для мостовых
   вызовов — то, что отдаёт мост; неизвестное помечается `n/a`, не выдумывается.
5. **Авторство фикстуры** (находка ревью v0.1, Notable): дефекты и баги воплощает
   Sonnet — кандидат в двух дисциплинах, а верифицирует Opus — инкумбент третьей.
   Общей памяти между контекстами нет, но «родной стиль» автора может делать
   дефекты статистически проще для моделей того же семейства. Смягчение: лид
   задаёт каждый дефект содержательно (§5); остаток асимметрии учитывается при
   интерпретации кросс-провайдерных счетов в RESULTS.md.
6. Степень OS-изоляции различается (Codex/Gemini — песочницы, Claude — нет);
   выравнивается отсутствием секретов на диске + сплошным аудитом журналов (§4).

## 11. Правила решений — предзафиксированы

1. Кандидат «прошёл дисциплину» = планка на **всех** задачах дисциплины.
2. Правка конфига **предлагается** (не вносится) когда: (а) боковой/дешёвый
   кандидат прошёл дисциплину, И (б) инкумбент не показал строго лучший счёт на
   тех же задачах ЛИБО ценовая разница существенна, И (в) телеметрия не
   противоречит. Форма правки — пилот: лейн получает реальный трафик под
   телеметрическим наблюдением (≥ 10 записей), закрепление — отдельным решением.
3. Провал инкумбента → сначала проверка валидности задачи: для дисциплин с
   верхним якорем — прошёл ли якорь; для architect (инкумбент — верхний тир) —
   не прошёл ли *нижний* якорь (§6.4). Если задача валидна — вопрос о поднятии
   маршрута, тем же порядком.
4. `never_shift` полигоном не пересматривается: первичный грейдер — Claude,
   judgment-класс — Fable, лестница эскалации — как есть.
5. Любая правка: бамп `version` в config.yaml + снапшот + строка в changelog
   README. Решение — совместно с пользователем; автоматических правок нет.
6. Результат одного полигона — *индикатор*. Две задачи на дисциплину не
   статистика; расхождение полигона с телеметрией трактуется в пользу телеметрии.
7. **Правило дисквалификации (аудит журналов).** Прогон дисквалифицируется, если
   журнал инструментов кандидата содержит обращение (чтение, листинг, выполнение)
   к любому из: `benchmark/**` (кроме выданного тикета), `hidden*`, истории git
   репозитория оркестрации, каталогу другого прогона `runs/**`, каталогам
   транскриптов `~/.claude/projects/**`. Дисквалификация — не FAIL планки, а
   отдельный исход `DISQUALIFIED` в scores.jsonl: она свидетельствует о поведении
   лейна и сама по себе — значимый результат для маршрутизации.
