# claude-orchestrate

Репозиторий скиллов оркестрации и полигона сравнения ролей (`benchmark/`).
Входов в оркестрацию два (ADR-0004, ADR-0006; третья голова `orca_orchestrate`
выведена, см. [ADR-0008](docs/adr/0008-retire-orca-transport.md)): полный
`/orchestrate` — сырые задачи и разбирательства, подготовительная фаза
(грилл → спека → тикеты) в его же сессии; тонкий `/orchestrate-frontier` —
исполнение готового фронтира тикетов на транспорте харнесса. Единое
исполнительное ядро под обоими — движок `skill/orchestrate/tools/run_lane/`
(транспорт; артефакт с диска; сверка модели). Этот файл читают все агент-CLI,
которым раздаётся работа — не только Claude Code, но и кросс-провайдерные
лейны (codex/grok/kimi/gemini), раздаваемые через `run-lane`.

## Agent skills

### Issue tracker

Задачи живут в GitHub Issues этого репозитория (`shah1git/claude-orchestrate`),
операции — через `gh` CLI. Нативные зависимости issue и sub-issues на репозитории
включены и проверены, поэтому карты `/wayfinder` и рёбра блокировки тикетов
выражаются машинно, а не текстовой пометкой в теле. См. `docs/agents/issue-tracker.md`.

### Triage labels

Пять канонических triage-ролей, строка метки совпадает с именем роли. Из них в
трекере пока заведена только `wontfix` (дефолтная гитхабовская); остальные четыре
создаются по мере первого применения. См. `docs/agents/triage-labels.md`.

### Domain docs

Single-context: `CONTEXT.md` и `docs/adr/` в корне репозитория. Сигналов монорепо
нет. См. `docs/agents/domain.md`.

## Доктрина RikaNV

Для внешнего читателя: каталог `doctrine/` — локальный клон приватного репозитория
`rikanv-doctrine`, в этом (публичном) репозитории отсутствует. Инструкции блока ниже
действуют в окружении владельца, где клон подключён; для знакомства с проектом снаружи
достаточно `README.md` и `docs/adr/`.

В проекте подключена доктрина `rikanv-doctrine` — единый источник инженерной
дисциплины RikaNV. Клон в `./doctrine/` (в `.gitignore`; маркер сверки
`.doctrine-version` — коммитится). Sync — только явной командой владельца:
`bash scripts/doctrine-sync.sh doctrine` (или агенту «синхронизируй доктрину»);
если сверка сообщила об изменении **hard**-файла — перечитать его перед задачей.

Этот репозиторий — **код-инструмент**, поэтому ведущий слой доктрины —
**инженерный**, не бренд-голос (публичного бренд-текста здесь нет; слой
`doctrine/voice/` применяется только если такой текст когда-нибудь появится).

Сверяться при любой не-косметической правке кода:

- [`doctrine/engineering/principles.md`](doctrine/engineering/principles.md) —
  архитектурная чистота (hard): архитектура до кода; контракт — обязательство;
  молчаливые отклонения запрещены (§2.2); **ограничение окружения не аргумент
  против корректности — менять окружение, а не огрублять алгоритм (§2.9)**;
  найденный баг закрывается от повтора регресс-стражем (§2.10); отказ от задачи
  лучше плохого решения (§2.8).
- [`doctrine/engineering/code.md`](doctrine/engineering/code.md) — нормы кода:
  модули, ошибки, документирование, тесты.
- [`doctrine/engineering/agent-reliability.md`](doctrine/engineering/agent-reliability.md)
  — доказательность сделанного (проверка не только выполнена, но и предъявлена),
  восстановление после паузы, калибровка уверенности.
- [`doctrine/engineering/agent-docs.md`](doctrine/engineering/agent-docs.md) —
  когда правишь `AGENTS.md`/`SKILL.md`/README/ADR: явность, разрешимые ссылки,
  понятность агенту любого вендора.

Приоритет при конфликте (ONBOARDING §3, правило 3): глобальный `~/.claude/CLAUDE.md`
задаёт нижнюю планку, `engineering/principles.md` её **ужесточает**; на любом
расхождении проектного правила с доктриной побеждает доктрина. Доктрину из этого
репозитория **не правят** — только PR в `rikanv-doctrine`.
