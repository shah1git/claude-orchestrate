# ADR-0004 — Фронтирный вход: отдельный тонкий скилл `/orchestrate-frontier`

- **Статус:** принято
- **Дата:** 2026-07-19
- **Решил:** владелец репозитория (сессия обсуждения 2026-07-19)
- **Затрагивает:** новый `skill/orchestrate-frontier/SKILL.md`;
  `skill/orchestrate/SKILL.md` (указатель в «The build spine»);
  `references/quality.md` §7 (поле `entry`); `install.sh`;
  `tools/validate_config.py` (охват соседних скиллов)

## Контекст

Хребет Покока (грилл → `to-spec` → `to-tickets`) вшит в `/orchestrate`
безусловно (ADR-0002). Реализация держит обе стороны шва «проработка ↔
исполнение» в одной сессии, из чего следуют три документированных адаптации:

1. обход `disable-model-invocation` — лид «читает файл и следует методу»
   вместо родного вызова скилла человеком;
2. слитый чекпойнт — Quiz-шаг `to-tickets` схлопнут с утверждением плана
   Step 1 в один Approve/Revise/Cancel с собственными правилами;
3. «умная зона» ~120k — хребет съедает 40–80k токенов окна лида до первого
   диспатча, и при подходе к границе предписан handoff в свежую сессию —
   то есть разрыв сессии уже существует как аварийный выход.

Одновременно два дефицитных ресурса сцеплены во времени: прожарке нужен
присутствующий владелец, вееру исполнителей — квота; при 3–4 параллельных
проектах недельное окно Claude исчерпывается за 1–2 дня до сброса
(standing mandate 2026-07-12), и требование «оба ресурса в момент старта»
регулярно недостижимо.

Сам ADR-0002 уже разрешил вопрос шва: «точка стыка — Тикет» (спека #4,
резолюция №1). Тикет по построению самодостаточен («размер — одно свежее
контекстное окно»), то есть интерфейс рассчитан на разрыв сессии.

## Решение

Вводится **второй вход** в оркестрацию — отдельный тонкий скилл
`/orchestrate-frontier`. Владелец проходит хребет сам, в сольной сессии,
родными вызовами скиллов Покока (`/grill-with-docs` → `/to-spec` →
`/to-tickets`; при тумане — `/wayfinder` до них), публикует тикеты в трекер
— и вызывает `/orchestrate-frontier`, который стартует сразу с
маршрутизации по готовому фронтиру.

Отдельный скилл, а не второй режим `/orchestrate`, по трём основаниям:

1. **Экономия окна исполнения.** SKILL.md `/orchestrate` — ~65 КБ (~16k
   токенов) и грузится целиком; фронтирному прогону грилл, хребет и
   развилка форм не нужны. Тонкий скилл возвращает исполнительной сессии
   ~10k+ токенов окна — ровно ту цель, ради которой затевался вынос.
2. **Чистый триггер вместо распознавания режима.** Владелец вызывает вход
   осознанно; внутри скилла нет ветвления «угадай, в каком я режиме».
3. **Пилот без риска для боевого v21:** `/orchestrate` не переписывается.

**Безусловность хребта (ADR-0002) сохраняется стражем провенанса:**
`/orchestrate-frontier` отказывается стартовать без опубликованной спеки и
восходящих к ней тикетов `ready-for-agent` — «сборку» без хребта он не
исполняет, он лишь не владеет хребтом (им владеет владелец). Сырые задачи
и «разбирательства» остаются целиком за `/orchestrate`.

**Правило нулевого дублирования.** Общая машинерия живёт в одном экземпляре:

| Что | Единственный источник |
|---|---|
| тюнинги (маршруты, гейт, лейны, пороги, бюджет) | `skill/orchestrate/config.yaml` |
| методика (delegation, quality, cross-provider, teams, clarify) | `skill/orchestrate/references/*.md` |
| телеметрия | `telemetry/routing-log.jsonl` + `tools/telemetry_append.py` |
| грилл, хребет, развилка форм | только `skill/orchestrate/SKILL.md` |
| входной протокол фронтира, карта цикла | только `skill/orchestrate-frontier/SKILL.md` |

Новый SKILL.md — входной протокол плюс краткие, насыщенные ссылками резюме
шагов 2–5; правило расхождения: при конфликте резюме с config.yaml или
reference-файлом правы последние, резюме чинится.

**Телеметрия — фальсифицируемый критерий пилота.** Каждая запись прогона
несёт `entry` ∈ `full | frontier` (§7; отсутствие = `full`, обратная
совместимость). После ~20 записей на вход сравниваются first-try pass rate
и ретраи/эскалации: если `frontier` не даёт как минимум паритета, посылка
«свежее окно исполнения покупает качество» опровергнута и скилл снимается.
Сравнение — только сопоставимого с сопоставимым: `frontier` против записей
`full` с `shape: сборка`, а не против всей популяции `full` (всякая запись
`frontier` — сборка по построению, поэтому примесь «разбирательств» в базе
сравнения измеряла бы форму работы, а не вход; уточнение внесено 2026-07-19
по вызову Spec-линзы к исходной формулировке).

## Последствия

- **Принято сознательно:** структурная гарантия «хребет пройден» слабеет до
  дисциплинарной на фронтирном входе — компенсируется стражем провенанса
  (без артефактов хребта в трекере вход не стартует).
- Ограниченное дублирование резюме-прозы шагов 2–5 признано и управляется
  правилом расхождения; значения и методика не дублируются нигде.
- **Именованное исключение — карта полного цикла.** Новый скилл открывается
  кратким описанием всех семи стадий (включая стадии подготовительной фазы,
  которыми он не владеет) и печатает её на экран при запуске. Это требование
  владельца (2026-07-19): вход, стартующий с середины процесса, обязан
  показать пользователю, где он находится и что предполагалось до него.
  Ориентирующая карта — не второй экземпляр метода: она перечисляет стадии и
  их следы, но не воспроизводит правил их исполнения, которые остаются в
  канонических файлах скиллов Покока и в полном `/orchestrate`.
- `install.sh` зеркалит новый скилл в `~/.claude/skills` и
  `~/.agents/skills`; `validate_config.py` расширяет охват прозы на
  `skill/*/SKILL.md`, чтобы ссылки нового файла на конфиг проверялись тем
  же валидатором.
- Чекпойнт нового входа сужен до маршрутизации (нарезка утверждена на
  Quiz-шаге `to-tickets` в сольной сессии) — особый случай «слитого
  чекпойнта» на этом входе исчезает.

## Отвергнутые альтернативы

- **Второй режим внутри `/orchestrate`** (аргумент или автоопределение):
  грузит те же 65 КБ, добавляет ветвление и риск ложного распознавания;
  выигрыша окна не даёт.
- **Вынести хребет и удалить его из `/orchestrate`:** ломает вход для
  компактных сборок и сырых задач; ADR-0002 требует, чтобы мимо хребта не
  проходило ничто, — встроенный путь остаётся для случаев, когда сольной
  проработки не было.
