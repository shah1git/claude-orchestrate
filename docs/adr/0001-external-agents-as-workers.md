# ADR-0001. Внешние агенты — воркеры/кросс-ревьюеры, а не второй оркестратор

- **Статус:** Принято 2026-06-19; **сохранён в этот репозиторий 2026-07-22** (см. рамку)
- **Дата:** 2026-06-19
- **Контекст-владелец:** dynamic workflow (Claude Code, `agent()`)

> **СОХРАНЕНО ИЗ РЕПОЗИТОРИЯ МОСТА (2026-07-22).** Этот ADR жил в отдельном
> проекте `/opt/tools/agent-bridge`, который снят с пути лейнов (ADR-0005 F7) и
> удалён. Его **МЕХАНИЗМ устарел**: «две поверхности» (мост `run-external-agent.mjs`
> + MCP `codex mcp-server`/`gemini-mcp-tool`) заменены единой консольной моделью —
> каждый внешний лейн вызывается своим vendor-CLI напрямую, форму строит исполнитель
> `tools/run-lane` (ADR-0005). Но его **РАССУЖДЕНИЕ — действующий канон**: внешние
> агенты остаются воркерами/кросс-ревьюерами, единственный владелец решений — лид;
> self-preference bias закрывается кросс-вендорной верификацией; гетерогенность
> оправдана маршрутизацией по стоимости и независимостью грейдера, а не разнообразием
> ради разнообразия. Читать ниже так: где сказано «мост»/«MCP» — подразумевается
> «run-lane + vendor-CLI»; всё про роли, независимость и обоснование — в силе.

## Контекст

Есть топовые подписки на **Codex** (ChatGPT) и **Gemini** (через `agy`/Antigravity), и желание
подключить их к нашему dynamic workflow. На рынке два класса решений:

1. **Готовые мульти-агентные оркестраторы** (pal/clink, ai-cli-mcp): сами раздают роли, делают
   `consensus`, ведут диалог.
2. **Лёгкие воркер-обёртки**: один внешний CLI = одна «рука», которой кто-то управляет.

Ключевое наблюдение: **наш `agent()`-workflow уже является оркестратором** — он декомпозирует,
делегирует, синтезирует. Это и есть «ведущий агент».

## Решение

Подключаем Codex и Gemini **как воркеров/кросс-ревьюеров**, а **единственным владельцем решений**
оставляем Claude-workflow. Конкретно:

- **Интерактив / ad-hoc** («спроси второе мнение») — через два лёгких MCP, уже подключённых в
  user-scope: нативный `codex mcp-server` и `gemini-mcp-tool` (backend `agy`).
- **Структурный вердикт в пайплайне** (машинно-проверяемый `APPROVED/WARNING/BLOCKED`) — через
  тонкий мост [`run-external-agent.mjs`](./run-external-agent.mjs): `codex exec --output-schema`
  и `agy --print`. Причина: у Codex `--output-schema` **не работает через MCP** (codex #15451),
  поэтому жёсткую схему даёт только прямой вызов.

**Готовый оркестратор (pal/clink) сознательно НЕ берём** в базовую схему.

## Обоснование (почему так, а не вторым оркестратором)

- **Дублирование.** Роли/`consensus`/continuation в pal повторяют то, что наш workflow и так делает —
  точнее и под себя.
- **Лишний слой и контекст.** Claude → pal → CLI длиннее, чем Claude → CLI; 18 инструментов pal
  раздувают контекст.
- **Сопровождение.** На этой машине нет старого `gemini` CLI (Google убрал 18.06), только `agy`;
  чтобы pal/clink работал с Gemini, нужен **патч в исходники** clink — его пришлось бы ребейзить на
  каждом апдейте. Мост же не патчит ничего.
- **Теория.** Anthropic («Building Effective Agents», «Multi-agent research system») + контр-взгляд
  Cognition («Don't Build Multi-Agents») + Стэнфорд (arXiv:2604.02460) сходятся: для **кода** держи
  **одного владельца решений**; передача задач между агентами теряет информацию и множит токены (~15×).
- **Где гетерогенность реально выигрывает** (а не «разнообразие ради разнообразия»):
  (1) **маршрутизация по стоимости** — дешёвое дешёвой модели; (2) **кросс-вендорная верификация** —
  у модели есть *self-preference bias* (предпочитает свои ответы), поэтому пусть **Codex/Gemini
  ревьюят диффы Claude**, а не Claude сам себя.

## Последствия

**Плюсы:** просто; ноль патчей; обе CLI автообновляются; обе поверхности уже подняты и протестированы
(`CLINK`-эквивалент не нужен); структурный вывод — через надёжный `--output-schema`.

**Минусы / принятые компромиссы:**
- Две поверхности вместо одной — но обе узкие. *(Устарело: v26 свёл к одной — vendor-CLI через run-lane.)*
- Нет встроенного `consensus`/голосования — при необходимости собираем в workflow сами (`Promise.all`
  двух вердиктов + сведение у Claude).
- Codex требует strict-схему — мост приводит автоматически. *(Теперь: `CodexAdapter.strictify_for_codex`.)*
- Gemini/agy без жёсткой схемы — парсинг best-effort; проверяем `ok`.

## Альтернатива в резерве

**pal/clink** как turnkey-оркестратор (роли, `consensus`, `codereview`) — оправдан, только если
понадобится «совет агентов» без написания кода. Тогда: установить в стабильный каталог, нести
agy-патч (`INTERNAL_DEFAULTS["agy"]` + `AgyTextParser` + `conf/cli_clients/agy.json`) и, по-хорошему,
оформить **upstream-PR** (находка общеполезна — Google убрал gemini CLI). Проверено рабочим
(`clink→codex` → `CLINK_CODEX_OK`, `clink→agy` → `CLINK_AGY_OK`).
