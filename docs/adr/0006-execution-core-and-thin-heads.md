# ADR-0006 — Единое исполнительное ядро и три тонкие головы (`orchestrate` / `orchestrate-frontier` / `orca_orchestrate`)

- **Статус:** принято
- **Дата:** 2026-07-22
- **Решил:** владелец репозитория (сессия обсуждения 2026-07-22)
- **Затрагивает:** `skill/orchestrate/SKILL.md` (Шаги 3–5 = исполнительное ядро);
  `skill/orchestrate-frontier/SKILL.md` (голова: ядро на харнесс-транспорте);
  новый `skill/orca_orchestrate/SKILL.md` (голова: ядро на Orca-транспорте);
  `skill/orchestrate/config.yaml` (поле `transport`, v26); `tools/validate_config.py`
  (охват соседних скиллов)
- **Расширяет:** [ADR-0004](0004-frontier-entry-skill.md) (фронтир как тонкий
  сиблинг) — распространяет тот же принцип на Orca-подложку; транспортный механизм —
  [ADR-0005](0005-lane-invocation-unification.md) / config v26

## Контекст

Есть (будет) три входа в оркестрацию, и все три сходятся в одном действии —
**готовый фронтир → исполнение**:

| вход | путь | подложка |
|---|---|---|
| `orchestrate` | сырая задача → хребет Покока → исполнение | харнесс |
| `orchestrate-frontier` | готовый фронтир → исполнение | харнесс |
| `orca_orchestrate` | готовый фронтир → исполнение | Orca |

Если исполнительная логика и настройки оркестрации живут в каждом входе отдельно —
они разойдутся. Это не абстрактный риск: ровно такое расхождение (форма вызова в
прозе разъехалась с действительностью) стоило дня 2026-07-21 и породило ADR-0005.
Настройки и метод исполнения обязаны быть единым источником для всех трёх.

## Что уже верно (не переделываем, фиксируем)

- **Настройки уже едины.** Маршрутизация (`routing.*`), гейт трёх линз (`gates.*`),
  цепочки замен (`availability.fallbacks`) — всё в одном `config.yaml`; каждый вход
  его читает. Единый источник настроек существует.
- **Ядро уже не дублируется, а ссылается.** `orchestrate-frontier/SKILL.md:187`
  дословно: «отсюда прогон ЕСТЬ Шаги 3–5 родственного плейбука, без сокращений»; его
  таблица (`:194–200`) отсылает к «sibling Step 2/3/4/5» — маршрут, параллелизм,
  гейты, синтез. Фронтир — уже тонкая голова над общим ядром.

Схема — достройка последнего шага, а не переделка.

## Решение

**Исполнительное ядро** = Шаги 3–5 плейбука `orchestrate` (маршрут · диспетч · гейт ·
синтез) + весь `config.yaml`. Это база. Три головы отличаются **ровно двумя
переключателями**:

```
        ┌─ ЯДРО: готовый фронтир → исполнение ─┐
        │  Шаги 3–5 (orchestrate) + config.yaml│
        └──────────────┬───────────────────────┘
                       │ используют (sibling-ссылка, не копия)
     ┌─────────────────┼──────────────────────┐
 orchestrate      orchestrate-frontier     orca_orchestrate
 хребет? ДА        хребет? НЕТ              хребет? НЕТ
 transport: харнесс transport: харнесс      transport: Orca (дефолт)
```

1. **Хребет до исполнения?** `orchestrate` гоняет хребет Покока (Шаги 0–1), затем
   ядро. Две другие головы входят прямо в ядро — фронтир уже проработан владельцем.
2. **Транспорт?** С v26 подложка — **одно поле `transport`** у лейнов, а не форк
   прозы. `orca_orchestrate` отличается от `orchestrate-frontier` **только** дефолтным
   транспортом (`orca-terminal`). Собственной исполнительной прозы у него нет — он
   ссылается на ядро тем же sibling-приёмом, что уже работает у фронтира.

Наименование `orca_orchestrate` — решение владельца 2026-07-22 (говорящее: что
оркеструет и на какой подложке).

## Топология ядра — решение, обоснованное доктриной, зафиксированное видимо

Ядро физически остаётся **внутри** `orchestrate/SKILL.md`, головы ссылаются на него
sibling-приёмом. Оно **не выносится** в отдельный базовый файл — **пока**.

Обоснование по доктрине RikaNV (`engineering/principles.md`, v2.20.0):

- **§1.7 (архитектура эволюционирует от сигналов износа).** Признаков износа сейчас
  нет: дублирования нет (ядро ссылается, не копируется), роста частных случаев в
  общем ядре нет, повторного обхода одного узла нет. Вынос обязателен **при сигнале**,
  а сигнала нет.
- **§2.4 (стоимость работы — не аргумент против чистоты).** Поэтому отсрочка
  обосновывается НЕ «дорого/лишняя работа» (это доктрина отвергает), а отсутствием
  сигнала износа по §1.7.
- **§1.6 / §2.2 (открытый вопрос — явно; молчаливых отклонений нет).** Отсрочка
  законна лишь как **видимое** решение с записанным триггером пересмотра — не как
  молчаливое «и так сойдёт». Триггер ниже.

**Триггер пересмотра (пересматриваем вынос ядра в первый же из случаев):**
- `orca_orchestrate` или будущая голова начинает плодить **транспорт-специфичные
  частные случаи** внутри общего ядра (§1.7: рост исключений);
- sibling-ссылка перестаёт чисто выражать голову (появляется «кроме Шага N, где…»);
- третья+ голова **дублирует** любую прозу ядра.

Первый такой сигнал → ядро выносится в первоклассную базу, и **все** головы (включая
сам `orchestrate`) ссылаются на неё симметрично.

## Инварианты (не меняются)

- Лид остаётся сессией-моделью (`never_shift`: оркестрационное суждение).
- `config.yaml` — единый источник настроек для всех трёх голов.
- Гейт трёх линз безусловен в каждой голове (ADR-0002); транспорт на глубину проверки
  не влияет (ADR-0005).
- Транспорт определяется по наличию: нет Orca → прямой vendor-CLI → пол «только
  Claude-работники». Пол не проседает.

## Открытый вопрос (зафиксирован явно, §1.6)

`orca_orchestrate` и `orchestrate-frontier` отличаются **только** дефолтным
транспортом. Сейчас это две именованные головы (решение владельца — говорящее имя).
Если появится **третий** транспорт, желающий свою голову, — это сигнал износа §1.7:
тогда головы схлопываются в одну фронтир-голову с аргументом `transport`, а не
плодятся по числу подложек. Записано как будущий триггер, решать при его наступлении.

## Следствия

- Три имени, но **одно исполнительное ядро и один набор настроек** — цель владельца
  «настройки оркестрации едины для всех трёх» достигнута структурно.
- Добавление `orca_orchestrate` — минимальная работа: тонкая голова + дефолтный
  транспорт, без второй прозы, которая могла бы разойтись.
- Решение прозрачно и записано; отсрочка выноса обоснована сигналом, а не удобством —
  «чистота как прозрачность» (доктрина, Главный принцип), а не «чистота как
  отсутствие работы».
