# ADR-0003 — Устойчивость оркестрации к выпадению целого провайдера

- **Статус:** принято (config v20, 2026-07-19)
- **Дата:** 2026-07-19
- **Решил:** владелец репозитория; проект — агент `architect` (Fable)
- **Затрагивает:** `skill/orchestrate/config.yaml` (блок `availability:`, `telemetry:`),
  `references/quality.md` (§7 словарь провайдеров), `references/cross-provider.md`
  (Runtime failure), `SKILL.md` (Step 2/4 — к синхронизации)

## Контекст

Устойчивость была **пофейсовой**: цепочка замен срабатывала, когда конкретный вызов
конкретного лейна не удавался. Провайдер, лёгший целиком, порождал серию одинаковых
отказов, каждый обрабатывался как первый — таймаут за таймаутом. Владелец поставил
вопрос шире: что делать при недоступности **целого провайдера**, в том числе
возникающей **посреди прогона**. Ключевое ограничение: оркестрация — не демон, а
лид-сессия; любой механизм ведёт лид на естественных точках (диспетч / возврат
воркера / граница волны), «тикать по таймеру» нечему.

## Решение

Пять механизмов (полный проект — прогон архитектора 2026-07-19; здесь — суть):

1. **Предохранитель по провайдеру** (`availability.breaker`). Единица — провайдер
   (`availability.provider_map` — единственное явное место принадлежности). Жёсткие
   сигналы (quota-window, subscription, connector-absent, hardening-gate) валят с
   первого события; мягкие (connector-error, timeout) — с двух отказов на **разных**
   диспетчах. Открылся → все ждущие лейны провайдера перебрасываются проактивно
   (`provider-degraded`), без контакта с мёртвым. Пробы — только на границе волны,
   ≤2 за прогон, первый боевой тикет назад — не-гейтовый (полуживой транспорт даёт
   ложный PASS). Второе открытие → списан до конца прогона. Между прогонами счётчик
   **не** переносится (в отличие от Orca — прогоны разделены часами): вместо этого
   прайор из телеметрии 24ч + обязательная канарейка.

2. **Карта «роль × упавший провайдер»** с инвариантом: каждая гейт-роль закрыта ≥2
   провайдерами при любом одиночном выпадении (после v18 пул это позволяет).
   `independence_filter: skip-producer-vendor` — критик не судит вендор продюсера.
   Закрыты реальные дыры v18/v19: критик/архитектор были одно-провайдерными,
   spec-линзы не было, gemini-critic висел в Standards вопреки блокировке.

3. **Протокол падения на лету** (`availability.midflight`). Смерть транспорта ≠
   провал качества (признак: есть ли пригодный отчёт). Смерть не жжёт
   `max_attempts_per_subtask`, но имеет свой предел (2 ре-диспетча). Ре-диспетч —
   свежий worktree + повторный stale-base guard; дифф покойника с диска — справка,
   не база (в базу — только именованным решением лида).

4. **Особый случай Claude** (`availability.lead`). Частичный выпад — карта (колонка
   anthropic). Смерть по квоте (частый случай) — прогноз на границе волны, при
   нехватке остатка окна плановый `stop-dispatch-finish-gates-report`. Полный выпад
   (лид мёртв) — честная граница: **оркестрировать некому, ждём возврата Claude**
   (решение владельца 2026-07-19); дисциплина resume-pack после каждой волны делает
   прогон резюмируемым. Аварийный чужой лид — `forbidden`; custodian-режим —
   `disabled` (владелец выбрал строгую паузу).

5. **Наблюдаемость** (`telemetry.provider_health_events`): события `provider_health`
   в общий поток + пер-тикет поля redispatch_of/salvage/run_id; починен словарь
   `provider` (был без xai/moonshot — записи grok/kimi врали о провайдере).

## Последствия

- Провайдер лёг → один переброс вместо N таймаутов; гейт качества не теряет
  независимости; прогон переживает частичный выпад Claude и резюмируется после
  полного. Числовые пороги — начальные догадки, правятся по данным (§5).
- Остаточный риск (назван честно): двойное сужение «одно выпадение +
  independence-фильтр» может у критика оставить единственного кандидата → тогда
  report-blocker, а не имитация гейта.
- Открытый вопрос владельцу закрыт: нет Claude — ждём (строгая пауза).

Связано: ADR-0001 (внешние агенты — воркеры, не второй оркестратор), ADR-0002
(хребет безусловен), config v18 (grok/kimi полные лейны), v19 (транспорт Gemini).
