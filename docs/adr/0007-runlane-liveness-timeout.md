# ADR-0007 — Исполнитель run-lane: ограничение по живости + записанная латентная огибающая, не по настенным часам

Статус: принято 2026-07-23. Основание — доктрина RikaNV `engineering/principles.md` §2.9 (ограничение окружения не аргумент против корректности), §2.10 (найденный баг закрывается регресс-стражем), §1.1/§1.2 (архитектура и value-объект до кода), §2.5 (каскад — одно атомарное изменение).

## Контекст

`run-lane` убивал лейн по НАСТЕННОМУ таймауту — полной длительности процесса:
`subprocess.run(timeout=…)` (`substrate.py`, SubprocessSubstrate), `budget_s = timeout`
(OrcaTerminalSubstrate), дефолт `DEFAULT_TIMEOUT_SECONDS = 300` (`__main__.py`), а на
крупных тикетах лид передавал 900с. Это **смешивало два разных понятия**: «процесс
завис» (мёртвый транспорт → убить) и «процесс тщательно думает» (жив, просто долго →
ждать).

Живой инцидент: лейн `kimi-k3` на max-эффорте — модель **корректна**, но медленна —
на крупном тикете упёрся в 900с и был помечен `transport-death`, работая верно. 900с —
**случайное** ограничение (произвольный дефолт), не требование. По §2.9 случайный лимит
права на код не имеет; убивать им корректную-но-медленную модель — это ровно
«огрубление окружением», которое §2.9 запрещает.

**Эмпирика (зонды run-lane, 2026-07-22/23).** Все четыре транспорта имеют тихое окно
рассуждения, масштабируемое с эффортом и размером тикета:

| Транспорт | Форма | Макс. тихая пауза | Живость во время работы |
|---|---|---|---|
| kimi (`stream-json`) | один блоб в конце | 32.4с (тривиально) → >900с (крупно) | нет — молчит до конца |
| grok (`json`) | один блоб | 16.1с | нет — молчит до конца |
| codex (`--json`) | ранние события + тихое рассуждение + ответ | 14.6с | частично (ранний пульс) |
| agy (`-p`) | инкрементальный текст | 6.4с (ведущая пауза) | да — стримит ответ |

Думание kimi/grok идёт **на сервере вендора**; локальный CLI лишь ждёт HTTP-ответа и
печатать в это время нечего. PTY/построчная буферизация окно рассуждения не устраняют
(модель молчит до первого токена). Значит **чистого idle-таймаута недостаточно**: для
«тихо думающих» лейнов idle вырождается в настенные часы.

Дополнительно: конверт исполнителя коллапсирует ЛЮБОЙ таймаут в `transport-death`
(`__main__._classify_run_result`), расходясь с собственным словарём маршрутизации
(`config.yaml` `midflight.death_evidence`), где `timeout-no-output` — смерть транспорта,
а таймаут С выводом (`ambiguity_resolves_to: quality`) — неоднозначность, ведомая дорогой
качества.

## Решение

Заменить единый настенный таймаут **двумя ортогональными пределами**, разделяющими
смешанные понятия:

1. **`idle_s` — предел живости** (инфраструктурный): максимум ТИШИНЫ (ни байта в
   stdout/stderr у subprocess; ни прироста файлов вывода у Orca). Пока сигнал идёт —
   процесс жив, сколько бы ни считал. Разумный глобальный дефолт-пол; детектор
   мёртвого/зависшего транспорта.
2. **`max_s` — латентная огибающая лейна** (доменное ТРЕБОВАНИЕ, §2.9): абсолютный
   потолок полной длительности, записанный **per-lane с обоснованием** из фактической
   латентности модели. Ловит «стримит, но зациклился». Не произвольный дефолт.

Оба нужны и ловят РАЗНЫЕ отказы: idle — прекращение вывода (зависание/обрыв); max —
бесконечный стриминг, при котором idle никогда не сработает.

Substrate ждёт по живости (приход вывода / прирост файла) и убивает только тишину сверх
`idle_s` или полную длительность сверх `max_s`. Решение о смерти — чистая функция
`classify_wait(now, last_output_at, started_at, limits) → continue | idle | envelope`
(инъекция времени, детерминированный тест). Контракт `Substrate.run(inv, limits)` несёт
иммутабельный value-объект `RunLimits(idle_s, max_s)` (§1.2). `RunResult.timeout_kind`
(`idle`/`envelope`) сопоставляется со словарём midflight. Дефолт `DEFAULT_TIMEOUT_SECONDS`
удалён — произвольного настенного дефолта больше нет.

**Регресс-страж уровня конфига (§2.10):** `validate_config` требует `latency_envelope` у
каждого кросс-провайдерного лейна — чтобы дефект не вернулся тихо через ненастроенный
лейн. Принятая деградация видима (тест + этот ADR), не молчалива.

## Отвергнутые альтернативы

- **Оставить настенные часы, поднять дефолт.** Отвергнуто по §2.9: выдаёт случайный
  лимит за требование; больший дефолт лишь отодвигает тот же дефект, продолжая срезать
  корректность модели ради инфраструктурной случайности.
- **«Снизить эффорт kimi» / «убрать kimi с больших задач».** Отвергнуто по §2.9 (и
  `code.md` §4.7): деформирует корректность (эффорт = тщательность рассуждения) под лимит
  платформы, либо обходит проблему. Меняем окружение (предел ожидания), не огрубляем
  алгоритм.
- **Чистый idle-таймаут без записанной огибающей.** Отвергнуто ЭМПИРИЧЕСКИ: kimi/grok
  молчат во время работы, чистый idle их убьёт или бесполезен; тихое окно рассуждения
  неустранимо (server-side reasoning; PTY не помогает).
- **PTY для принудительной построчной буферизации.** Отвергнуто: меняет формат вывода
  вендора (риск JSON-парсинга), фрагильно, и не трогает окно рассуждения.
- **Больший per-lane настенный дефолт вместо огибающей.** Отвергнуто по §2.9, гейт (2):
  случайное, не классифицированное осознанно как требование.

## Последствия

Корректность модели больше не разменивается на инфраструктурный лимит. Контракт
`Substrate.run` меняется на value-объект `RunLimits` — каскад по всем вызывающим (ABC,
две подложки, `__main__`, `detect`, `smoke`, адаптерный agy `--print-timeout`, тесты) как
одно атомарное изменение (§2.5). Появляется требование записывать латентную огибающую
каждого кросс-провайдерного лейна (видимая принятая деградация, §2.9/§2.10). Классификация
таймаутов обогащается `timeout_kind` и сходится со словарём midflight (idle →
`timeout-no-output`; envelope-с-выводом → неоднозначность, дорога качества).

Стартовые значения (тюнинг владельца по телеметрии): `idle_default_s: 300`,
`max_default_s: null`; `kimi-k3.latency_envelope: {idle_s: 1800, max_s: 2400}` с
обоснованием из замеров.

Реализация — срезами S0…S5 (ядро политики → SubprocessSubstrate → OrcaTerminalSubstrate →
конфиг+валидатор → проводка CLI/detect/smoke → проза), каждый с приёмкой и регресс-стражем.
